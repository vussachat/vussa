# Full-Featured, Highly Available Team Chat

## Summary

Implement the work in this order:

1. Refactor the oversized backend `main.rs` into maintainable modules.
2. Establish comprehensive testing infrastructure.
3. Add messaging, search, files, notifications, profiles, and moderation.
4. Harden the system for Kubernetes-native HA and fail-over.
5. Validate every function, API, event, migration, and operational path.

## Phase 1: Backend refactoring

Split `backend/src/main.rs` into focused modules while preserving behavior and API compatibility.

Recommended structure:

- `config.rs` — environment configuration and defaults
- `error.rs` — application and repository errors
- `models.rs` — database, API, WebSocket, and domain types
- `auth/` — registration, login, sessions, CSRF, password verification
- `repository/` — repository traits and PostgreSQL implementations
- `api/` — HTTP handlers grouped by authentication, account, channels, users, and administration
- `websocket/` — connection lifecycle, client events, server events, room handling, reconnection support
- `messages/` — sending, editing, deletion, history, pagination, and caching
- `presence/` — presence, typing, status, and participant synchronization
- `notifications/` — notification creation, delivery, preferences, and read state
- `search/` — message indexing and search handlers
- `storage/` — file metadata and object-storage integration
- `outbox.rs` — transactional outbox publishing and retries
- `metrics.rs` — metrics and operational endpoints
- `state.rs` — shared application state and dependency construction
- `routes.rs` — router assembly
- `main.rs` — process startup only

Refactoring requirements:

- Preserve existing routes, event names, permissions, and database behavior.
- Introduce explicit traits around PostgreSQL, Valkey, object storage, notifications, and clock/time providers.
- Remove duplicated authorization, validation, serialization, and error-mapping logic.
- Keep modules independently testable.
- Add characterization tests before moving behavior.
- Compile and run the existing test/benchmark flows after each refactoring stage.
- Avoid changing public behavior until the modular structure is stable.

## Phase 2: Testing foundation

Before adding features:

- Add unit-test modules beside each backend module.
- Add PostgreSQL repository integration tests.
- Add Valkey integration tests.
- Add HTTP handler tests.
- Add WebSocket protocol tests.
- Add frontend component and state-reducer tests.
- Add end-to-end test fixtures with isolated databases and cache namespaces.
- Add deterministic clock and ID providers for tests.
- Add failure-injection interfaces for database, cache, storage, and notification failures.
- Add CI checks for formatting, static analysis, backend tests, frontend checks, and migrations.

Every existing function should receive either a direct unit test, an integration test, an end-to-end test, or an explicit documented reason why direct testing is inappropriate.

## Phase 3: Core chat capabilities

Add and test:

- Message deletion and restoration
- Reactions
- Threads and replies
- Message permalinks
- Markdown and safe link rendering
- User, channel, and online-user mentions
- Link previews
- Draft persistence
- Idempotent message sending
- Message reports
- Realtime deletion, reaction, and thread events

Extend message data with deletion state, root-message references, reply counts, structured metadata, mentions, edit history, and idempotency keys.

## Phase 4: Search, unread state, and notifications

Add full-text search, filters, pagination, highlighted results, saved messages, favorites, recent conversations, read cursors, unread counts, typing indicators, user status, notification preferences, browser push, email notification adapters, and multi-device synchronization.

Use PostgreSQL for durable search, read state, and notification state. Use Valkey TTL keys for ephemeral presence and typing.

## Phase 5: Files and media

Add multipart uploads, object-storage abstraction, attachment metadata, authorized downloads, previews, validation, checksums, scanning hooks, retention cleanup, and orphaned-upload cleanup.

## Phase 6: Channels, users, and moderation

Add channel settings, join/leave, owners, moderators, invite links, favorites, archived-channel browsing, system messages, profiles, custom statuses, account recovery, session management, account export/deletion, bans, moderation queues, posting restrictions, spam controls, and complete audit coverage.

## Phase 7: Kubernetes-native HA and fail-over

Keep backend replicas stateless behind a Kubernetes Service. Add Helm support for replicas, rolling updates, PodDisruptionBudget, HPA, topology spread, anti-affinity, resources, probes, graceful termination, Service/Ingress, NetworkPolicy, metrics, external secrets, migration Jobs, and backup/restore Jobs.

Use HA-capable external services or operators for PostgreSQL and Valkey. Support and test pod termination, rolling deployments, database/cache fail-over, dependency outages, outbox replay, duplicate/delayed events, reconnect storms, node/zone failure, graceful shutdown, and recovery without message loss or privilege escalation.

## Required validation

Test backend units, repositories, handlers, WebSockets, migrations, properties, concurrency, security, load, and failure injection. Test frontend components, reducers, reconnects, permissions, uploads, accessibility, and responsive layouts. Validate Helm, manifests, containers, multi-replica behavior, rolling updates, chaos, backup/restore, alerts, and migration compatibility.

CI must fail if formatting, static analysis, any test suite, migration validation, frontend checks, container builds, or HA smoke tests fail.

## Assumptions

- `main.rs` refactoring is the first implementation milestone.
- Existing routes, WebSocket events, permissions, and migrations remain compatible during refactoring.
- PostgreSQL is the durable source of truth.
- Valkey handles shared realtime coordination and ephemeral state.
- Object storage holds file contents.
- No feature is complete until normal, unauthorized, concurrent, failure, and recovery paths are tested.
- Rolling upgrades and backward-compatible migrations are required.
